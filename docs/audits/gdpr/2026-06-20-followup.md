## Evidence

### F-01/F-02 CORS abuse on contact + feedback functions (High → Closed)
```
POST /functions/v1/send-contact-email     Origin: https://evil.example.com → HTTP 403 {"error":"Origin not allowed"}
POST /functions/v1/submit-product-feedback Origin: https://evil.example.com → HTTP 403 {"error":"Origin not allowed"}
OPTIONS … Origin: https://dlinrt.eu        → 200, ACAO: https://dlinrt.eu (allowlist enforced)
```
Both functions reject non-allowlisted origins *before* parsing the body via the shared `rejectDisallowedOrigin` helper in `supabase/functions/_shared/cors.ts`.

### F-03 Privacy policy Art. 13/14 gaps (High → Closed)
`src/pages/PrivacyPolicy.tsx` now discloses:
- 13 data categories with purpose, lawful basis, retention period
- Sub-processors page link + SCC transfer mechanism
- Concrete DSR UX paths (Profile → Data Export, Delete Account, Cookie Settings, unsubscribe)
- "Last updated: June 20, 2026"
- Statement on no special-category data and no Art. 22 automated decisions

`/subprocessors` (HTTP 200) lists Supabase, Resend, Mailchimp, Paddle, GitHub, Lovable with location and safeguard.
(Note: grep against the prod URL returns 0 hits because the SPA renders these terms client-side; verified by source review of `PrivacyPolicy.tsx` and `Subprocessors.tsx`.)

### F-04 Retention not enforced (High → Closed)
- Edge function `gdpr-retention-purge` deletes per-table beyond promised TTLs (analytics 365d, security_events 180d, mfa_activity_log 365d, email_send_log 90d, contact_submissions 730d, admin_audit_log 730d).
- Migration `20260620182800_…sql` enables `pg_cron` + `pg_net` and schedules the function nightly at 03:00 UTC.
- Manual invocation succeeded in prior run; counts written to `admin_audit_log`.

### F-05 delete-account leaves email-keyed PII (High → Closed)
`supabase/functions/delete-account/index.ts` now scrubs `newsletter_subscribers`, `contact_submissions`, `product_feedback`, `email_send_log` by replacing recipient/submitter email with `deleted-<sha256(email)>@redacted.local` and nulling free-text fields before removing the auth user.

### F-06 RLS posture on PII tables (re-verified Closed)
Anonymous REST `SELECT` returns 401 on every PII table:
```
profiles, contact_submissions, newsletter_subscribers, product_feedback,
security_events, mfa_activity_log, email_send_log, consent_audit_log, user_roles → all 401
```

### F-07 No self-service portability beyond profile (Medium → Closed)
- New edge function `gdpr-data-export` aggregates 15 PII categories (incl. service-role-only tables: security_events, email_send_log, mfa_activity_log, consent_audit_log).
- Unauthenticated call → `HTTP 401 {"error":"Missing authorization"}`.
- Wired into `src/components/profile/DataExport.tsx`.

### F-08 Missing sub-processor disclosure (High → Closed)
`/subprocessors` page live; footer link present.

### F-09 IP handling (re-verified Closed)
Code review: all writers (`track-security-event`, `track-analytics`, `subscribe-newsletter`, `send-contact-email`) hash IP via SHA-256 before storage. No raw IPs stored.

### F-10 MFA backup codes & unsubscribe tokens (re-verified Closed)
Bcrypt for backup codes; HMAC-SHA256 unsubscribe tokens unchanged.

### F-11 Rate limiting on public edge functions (Medium → **Deferred**)
Not yet implemented. Current mitigation: Supabase platform-level limits + CORS allowlist + size limits. Recommend Postgres-backed `check_rate_limit` RPC + 5-function rollout per original plan. Tracked.

### F-12 Production security headers (Low → **Partial**)
`public/_headers` defines a strict CSP, `Cross-Origin-Opener-Policy: same-origin`, `Permissions-Policy`, `X-Content-Type-Options: nosniff`, `Referrer-Policy`. Live response from `https://dlinrt.eu/` shows HSTS, `Referrer-Policy`, `X-Content-Type-Options` — but CSP / COOP / Permissions-Policy headers are stripped by the Lovable custom-domain proxy. Lovable preview/published URLs honor them. Action: file infra request to forward `_headers` directives on custom-domain edge.

## Conclusion
Waves A, B, D fully closed. Wave C closed for CORS hardening; rate-limiting and header propagation deferred with clear next steps. No High-severity GDPR findings remain open. The site materially meets Art. 13/14 transparency, Art. 15/20 portability, Art. 17 erasure (with email scrubbing), and Art. 5(1)(e) storage-limitation obligations.
