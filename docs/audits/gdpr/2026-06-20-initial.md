# GDPR Penetration Audit — DLinRT.eu
**Date:** 2026-06-20 · **Scope:** production app, Supabase project `msyfxyxzjyowwasgturs`, edge functions, public site
**Method:** read-only code review + live black-box probes (anon PostgREST and edge functions) from sandbox. No data was exfiltrated; no real user records were modified.

---

## Executive summary

Overall posture: **Good technical baseline, weak legal/transparency layer.**

| Area | Rating | Notes |
|---|---|---|
| RLS / table access control | 🟢 Strong | Every PII table denies anon SELECT/INSERT; admin/reviewer reads gated by `has_role()`. |
| Edge-function auth & rate-limit | 🟡 Mostly OK | Per-function CORS allowlist + in-memory rate limit + auth where required. One CSRF-like gap (CORS not enforced on POST). |
| Auth lifecycle (MFA, deletion) | 🟢 Strong | MFA backup codes bcrypt-hashed; delete-account edge function with password + MFA re-auth; HMAC-signed unsubscribe tokens. |
| Transport / headers | 🟢 Strong | HSTS preload, CSP, COOP/COEP/CORP, Permissions-Policy locked down. |
| Cookie consent | 🟢 Strong | Analytics gated behind explicit opt-in; Secure + SameSite=Strict; reload after grant. |
| **Privacy Policy (Art. 13/14)** | 🔴 **Materially incomplete** | Stale (Dec 2024), undisclosed sub-processors (Resend, Mailchimp, Paddle, GitHub, Lovable), undisclosed US transfers, missing data-controller identity, DPO contact, right-to-complain (Art. 13(2)(d)), portability/restriction/object. |
| **Data inventory vs. policy** | 🔴 **Mismatch** | Policy claims only newsletter + contact data is collected, but DB actually stores profiles, MFA factors, roles, consent log, security events, analytics visitors, product feedback, user_products, registration notifications, email send log. |
| Sub-processors & int'l transfers | 🔴 **Undisclosed** | Resend is US-based and receives email + name; no SCC reference; not named in policy. |
| Retention / erasure automation | 🟡 Partial | `delete-account` exists; no scheduled purges for `contact_submissions` (claimed ≤2y), `analytics_*` (claimed 1y), `security_events`, `email_send_log`. |

**12 findings**: 0 critical · 5 high · 4 medium · 3 low.

## Track 1 — Data inventory vs. lawful basis (Art. 5, 6, 13, 30)

### F-01 [HIGH · Art. 13(1)(c), 30] Privacy Policy under-discloses data categories
The policy at `src/pages/PrivacyPolicy.tsx` enumerates only *"first name, last name, email, message, timestamp, consent records"*. The database actually persists the following PII not mentioned in the policy:

| Table | PII not disclosed in policy |
|---|---|
| `profiles` | institution, specialization, bio, profile_image_url, linkedin_url, MFA enrolment/backup-code timestamps, approval status, data-processing-consent fields |
| `consent_audit_log` | ip_hash, user_agent, action history |
| `mfa_activity_log` / `mfa_backup_codes` | factor type, ip_hash, hashed backup codes |
| `security_events` | ip_hash, user_agent_hash, url, severity |
| `analytics_visitors` / `analytics_page_visits` | visitor_id cookie, path, duration |
| `user_products` | institution, department, experience notes, use case |
| `product_feedback` | submitter name & email, supporting URL |
| `user_registration_notifications` | email, verification status |
| `email_send_log` | recipient address, bounce metadata |
| `contact_submissions` | submission_method, status |
| `role_change_log`, `role_requests` | user_id, requested_role, justification |

**Recommendation:** rewrite the *What Data We Collect* and *Records of Processing* sections to enumerate every category, separated by user-type (anonymous visitor, newsletter subscriber, registered user, reviewer, company representative).

### F-02 [HIGH · Art. 13(2)(a-d), 14] Missing required Art. 13/14 disclosures
The policy omits:
- Identity & contact details of the **controller** (only `info@dlinrt.eu`; no legal entity, address, country).
- **DPO / privacy contact** — only a generic mailbox.
- **Retention periods per category** (only newsletter + contact + analytics covered; no entry for profiles, MFA, security_events, audit logs, role data, email_send_log, product_feedback).
- **Right to lodge a complaint** with a supervisory authority (Art. 77).
- **Right to data portability** (Art. 20), **restriction** (Art. 18), **object** (Art. 21), **not be subject to solely-automated decisions** (Art. 22).
- **Legal basis** declaration per processing activity (Art. 6(1)(a/b/c/f)).
- **Whether the data subject is required to provide the data** and consequences of refusal (Art. 13(2)(e)).
- **Existence of automated decision-making / profiling** (Art. 13(2)(f)) — relevant for analytics + role approval workflow.

### F-03 [HIGH · Art. 13(1)(f), 28, 44, 46] Undisclosed sub-processors & international transfers
Code calls these sub-processors that the policy does not name:

| Sub-processor | Purpose | Hosting | Disclosed? | Risk |
|---|---|---|---|---|
| **Resend** (`npm:resend@4.0.0` in 12+ edge functions) | Transactional + newsletter email | **US** | ❌ "secure, GDPR-compliant email service provider" only | Art. 13(1)(f), Art. 44 — US transfer without naming SCC / Adequacy (DPF). |
| **Mailchimp** (admin-newsletter-management, sync-newsletter-audience) | Newsletter audience sync | **US** | ❌ | Same as above. |
| **Paddle** (donation system per memory) | Donation processing (MoR) | UK/US/IE | ❌ | Likely controller-to-controller, still needs disclosure. |
| **GitHub** (apply-product-edit, github-pull-requests, invite flows) | PR creation, file checks | US | ❌ | Account email of editor potentially exposed. |
| **Supabase** | Auth + DB + Edge Functions | Region not stated in policy | Policy claims "EU-hosted" | Verify the project is actually `eu-*` — if `us-*`, statement is false. |
| **Lovable / Lovable Cloud** | App hosting & CI | Mixed | ❌ | Disclosure recommended. |

**Recommendation:** publish a `/subprocessors` page listing each entity, purpose, country, transfer mechanism (SCC, DPF), and update the policy to reference it. Verify the Supabase project region (the project ref `msyfxyxzjyowwasgturs` should be cross-checked in the Supabase dashboard).

### F-04 [MEDIUM · Art. 5(1)(b)] Stale "Last updated" date
Policy header reads *"Last updated: December 17, 2024"*. Since then the schema has grown (MFA, consent_audit_log, security_events, analytics, role workflow, donations, certifications). The policy must be re-dated and re-versioned on every material change.

## Track 2 — RLS pen-test (Art. 5(1)(f), 32)

Live probes from an anonymous client (anon JWT) against `/rest/v1/<table>?select=*&limit=1`:

```
profiles, contact_submissions, newsletter_subscribers, user_products,
product_feedback, security_events, analytics_visitors, analytics_page_visits,
mfa_backup_codes, mfa_activity_log, consent_audit_log, email_send_log,
admin_audit_log, user_registration_notifications, role_change_log,
role_requests, user_roles
  → ALL returned HTTP 401 "permission denied for table"
```

Anon INSERT probes against `contact_submissions`, `newsletter_subscribers`, `product_feedback`, `security_events`, `user_products`: **all 401**. Writes correctly forced through edge functions with service role.

### F-05 [INFO] RLS hardening is strong
The "Deny anonymous access to X" + per-role `has_role()` policy pattern is applied consistently. No `USING(true)` policies were found on sensitive tables. **No anon read or write path was discoverable.** ✅

### F-06 [LOW · Art. 32] `consent_audit_log` and `email_send_log` SELECT policies are scoped to role `public` instead of `authenticated`
Functionally equivalent because `has_role(auth.uid(),'admin')` fails for anon (uid is null), but the role grant pattern is inconsistent with the rest of the schema. Tighten to `TO authenticated` for defence-in-depth.

## Track 3 — Edge function abuse testing (Art. 25, 32)

Probes from an unauthenticated client with `Origin: https://evil.example.com`:

| Function | Status | ACAO header | Notes |
|---|---|---|---|
| subscribe-newsletter | 400 | absent | Rejected (validation) |
| send-contact-email | **200** | absent | **Processed despite forbidden origin** — see F-07 |
| submit-product-feedback | 400 | absent | Rejected (validation) |
| track-analytics | 403 | absent | Origin allowlist enforced server-side ✅ |
| track-security-event | 401 | absent | Requires auth ✅ |
| unsubscribe-newsletter | 400 | absent | Rejected (token validation) ✅ |

### F-07 [HIGH · Art. 32, OWASP CSRF] `send-contact-email` and `submit-product-feedback` do not server-side enforce the Origin allowlist
The `getCorsHeaders()` helper returns a "safe default" ACAO when the Origin is not allowlisted, but the handlers **still execute the request body**. Browsers will block the response, but cURL / server-to-server / mobile-app callers (and bots) are unaffected. Combined with email sending, this enables anonymous spam relay through the dlinrt.eu sender domain at the rate-limit cap (5 contact emails per 5 min per IP, 10 feedback / 15 min per IP — abusable from a botnet).

**Recommendation:** mirror the pattern from `track-analytics` (early 403 when origin is not allowlisted) in all public POST handlers, and also reject when `Origin` header is absent for non-OPTIONS POST.

### F-08 [MEDIUM · Art. 32] In-memory rate-limit is per-instance and resets on cold start
Supabase Edge Functions scale horizontally; `new Map()` is local to one isolate. An attacker who hits different isolates (load-balanced) bypasses the cap. Burst of 15 requests in our test all returned 400 *for the wrong field-name reason*, so the limiter was never reached — meaning we cannot confirm enforcement either. **Move rate-limit state to Postgres (e.g. `rate_limit_buckets(ip_hash, function, window_start, count)`)** with a unique index + upsert.

### F-09 [LOW · Art. 32] No CAPTCHA on public POSTs
Contact, newsletter, feedback all accept raw POSTs. Combined with F-08, this is exploitable for spam. Add Cloudflare Turnstile or hCaptcha invisible challenge.

## Track 4 — Authentication & account lifecycle (Art. 32)

### F-10 [INFO] Account-deletion path correctly purges PII
`delete-account` requires password + MFA re-verification, calls admin client to delete the auth user; cascading FKs on `profiles`, `user_roles`, `user_products`, `mfa_backup_codes`, `mfa_activity_log`, `consent_audit_log`, `role_requests`, `role_change_log` handle erasure. ✅

**Verify:** that `contact_submissions`, `product_feedback`, `newsletter_subscribers`, `email_send_log` are scrubbed for the user's email address on deletion. These tables don't have user_id FKs (PII keyed by email) and likely survive deletion — Art. 17 conflict. See F-12.

### F-11 [INFO] MFA backup codes hashed with bcrypt (`store-backup-code`), unsubscribe tokens are HMAC-SHA256 with TTL ✅

### F-12 [HIGH · Art. 17] No automated retention / scheduled erasure
- `contact_submissions`: policy promises "typically not exceeding 2 years" — no cron found in `supabase/functions/` that deletes them.
- `analytics_visitors`, `analytics_page_visits`: policy promises 1-year deletion — no cron.
- `security_events`, `email_send_log`, `admin_audit_log`, `mfa_activity_log`, `assignment_history`: no documented retention.
- After `delete-account`, rows in the above tables keyed by **email or ip_hash** survive forever.

**Recommendation:** add a scheduled `cleanup-retention` edge function (Supabase cron) that runs daily and deletes rows past their published retention. On account deletion, also scrub `newsletter_subscribers`, `contact_submissions`, `product_feedback` rows matching the user's email.

## Track 6 — Transport, headers, cookies (Art. 32)

`public/_headers`:
- HSTS `max-age=31536000; includeSubDomains; preload` ✅
- CSP present; `script-src` allows `'unsafe-inline'` and `https://cdn.gpteng.co` (Lovable badge loader) — acceptable but tighten when published.
- `img-src 'self' data: blob: https:` — broad, allows any HTTPS image (used for product logos via `https:` URLs in product data).
- COOP/COEP/CORP set to same-origin / require-corp ✅
- Permissions-Policy: camera, mic, geo, payment, etc. all disabled ✅

Cookies (`src/utils/cookieUtils.ts`):
- Consent cookie, visitor-id cookie set with `SameSite=Strict; Secure` (dev-only excluded) ✅
- HttpOnly intentionally not set (documented; necessary for the JS banner) — acceptable.
- Analytics cookies gated through `getCookieConsent()` check before write ✅.

### F-14 [LOW · Art. 32] CSP allows `script-src 'unsafe-inline'`
Replace inline script and inline-style usage with nonces or hashes, then drop `'unsafe-inline'` from `script-src`. Lower priority because there is no user-generated HTML rendered in the app.

## Track 7 — PII leakage surface

- `security_events`, `mfa_activity_log`, `consent_audit_log` store **ip_hash** + **user_agent / user_agent_hash** (per Core memory rule). ✅
- `admin_audit_log.ip_address` is `text` — **not hashed**. Confirm whether the edge functions that write to it hash before insert. If raw IP is stored, this contradicts the Core security rule.

### F-15 [MEDIUM · Art. 5(1)(c), security memory] `admin_audit_log.ip_address` column allows raw IP
Either rename to `ip_hash` and store SHA-256, or document an exception (e.g., for forensic admin actions retained 90 days). Audit recent rows: ```SELECT ip_address FROM admin_audit_log ORDER BY created_at DESC LIMIT 10``` (admin-only) to verify format.
